## 一、项目概览

### 1.1 一句话定位

**别的产品是「人→内容→人」，Standby 是「人→事物→感受→人」。**

### 1.2 核心理念

- **心物驱动**：没有心物，就没有表达
- **感受链**：替代传统的「内容+评论」模式
- **匿名架构**：攻击在架构层面就难以组织
- **共鸣过滤**：不是「你可能喜欢」，而是「你们确实有共鸣」
- **五层安全**：硬件身份、端到端通信、盲服务器、完全匿名、群体监测
- **真实需求数据**：用户表达聚合为群体共识，为供给侧提供真实需求

### 1.3 目标用户

大城市原子化个体——在大城市中感到孤独、渴望真实连接但抗拒传统社交方式的人。

## 三、核心概念速查

| 概念 | 定义 | 关键特性 |
|------|------|---------|
| **心物** | 触发感受的客观事物 | 完整、具体、真实 |
| **感受链** | 表达形成的链式反应 | 树状结构、独立表达 |
| **共鸣** | 对同一心物的相似情感反应 | 不是点赞，是「我也有这种感觉」 |
| **知己** | 通过持续共鸣自然沉淀的关系 | 感受的连接，不是身份的连接 |
| **匿名身份** | 用户在系统中的匿名展示 | 随机昵称、随机头像、无个人主页 |

## 七、设计原则速查

### 7.1 产品设计原则

1. **心物驱动，非内容驱动**
2. **感受链，非评论区**
3. **匿名架构，安全优先**
4. **共鸣过滤，非算法推荐**
5. **深度优先，非正负优先**
6. **沉默表达，非公开表演**
7. **沉淀关系，非积累人脉**

### 7.2 技术设计原则

1. **隐私安全第一**
2. **最低开销**
3. **端侧优先**
4. **AI 高兼容**
5. **独立部署**

### 7.3 UI 设计原则

1. **无感的、融入生活的**
2. **真实 > 流量**
3. **先共享体验，再发现人**
4. **渐进增强，非功能解锁**
5. **匿名性永不打破**

## 八、常见问题

### 8.1 为什么没有评论区？

传统评论区是「广播+回应」模式，Standby 使用「感受链」替代。每条表达都是独立的，不是「回复」别人，而是「被触发」的新感受。

### 8.2 为什么不能搜索用户？

为了保护匿名性。用户不能被搜索、不能被追踪、不能被骚扰。知己关系是通过共鸣自然涌现的，不是通过搜索找到的。

### 8.3 为什么没有点赞数？

点赞数会引发表演动机。Standby 只有「共鸣」——「我也有这种感觉」，不是「我喜欢这个」。

### 8.4 知己关系如何建立？

通过持续共鸣自然沉淀。系统检测到两个人在多个心物上反复共鸣，会自动提示「有个人总是跟你有同样的感受」。双方都选择「想认识这个人」后，可以互相看到设定昵称和头像。

## 九、决策记录

| # | 决策 | 理由 |
|---|------|------|
| 1 | 心物由平台+用户+AI三种来源 | 冷启动+原生多元+有机循环 |
| 2 | 强实名注册+全匿名展示 | 守住底线+消除表演压力 |
| 3 | 无个人主页、不可搜索用户 | 避免骚扰，保护匿名安全感 |
| 4 | 知己是感受知己，不是身份知己 | 匿名性永不打破 |
| 5 | 关系深度是光谱，不是阶段 | 用户永远看不到数字 |
| 6 | 升级/降级是自然发生的 | 没有断开连接的操作 |
| 7 | 个性化基线+相对共鸣强度 | 尊重每个人的互动节奏 |
| 8 | 关系暂停而非衰减 | 跨时空互动不惩罚沉默 |
| 9 | 功能渐进解锁 | 知己不对新用户展示 |
| 10 | 知己入口在「我」页面内 | 不是独立 Tab |
| 11 | 五层安全体系 | 硬件身份、端到端通信、盲服务器、完全匿名、群体监测 |
| 12 | 真实需求数据变现 | 用户表达聚合为群体共识，为供给侧提供真实需求 |
| 13 | 内啡肽驱动 | 满足感（成长、共鸣、发现、创造、被看见）vs 多巴胺（想要） |
