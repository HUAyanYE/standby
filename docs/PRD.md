# Standby — 产品需求文档 v0.1

> 面向 AI 时代人与人交互的跨端跨设备新一代应用。

---

## 一、产品愿景

### 1.1 核心命题

**在 AI 让一切都可以被伪造的时代，如何重建人与人之间的真实连接？**

Standby 不是一个社交平台，不是一个内容分发系统。它是一个——让人重新敢于表达真实自我的安全空间。

### 1.2 观察到的时代矛盾

```
技术能力 ↑  →  AI 可以生成一切  →  真假难辨  →  信任 ↓
效率工具 ↑  →  流量至上逻辑    →  对立/虚伪 ↑  →  真实表达 ↓
```

结果：人们有能力表达，但不敢表达；有渠道交流，但不愿交流。

流量至上的内容生态催生对立、骂战、虚伪。高效的 AI 生成工具进一步放大了这一问题。人们不再愿意表达真实看法、交流真挚感情。

### 1.3 设计原则

| 维度 | Standby 的选择 | 对标现状 |
|---|---|---|
| **内容逻辑** | 真实 > 流量 | 流量 > 真实 |
| **AI 角色** | 守护者 + 织网者（过滤不良、编织共鸣） | 助推者（放大内容、优化传播） |
| **交互形态** | 无感的、融入生活的 | 刻意的、打开 App 才存在的 |
| **设备关系** | 系统级输入，不分设备 | App 级隔离 |
| **关系建立** | 先共享体验，再发现人 | 先关注人，再看内容 |

好的人与人之间的交流是**无感的、真诚的、积极的**。

## 二、目标用户

### 2.1 主要用户画像

**大城市原子化个体**：
- 一人在大城市工作生活，家人朋友不在身边
- 渴望真挚且真实的交流、沟通和共鸣
- 对现有社交平台失望（表演压力、流量逻辑、虚假内容）
- 有表达欲望但缺乏安全的表达空间

### 2.2 用户核心需求

- 被理解（而非被看见）
- 找到共鸣（而非获取关注）
- 真实连接（而非社交表演）

## 三、核心概念

### 3.1 锚点模型（Anchor Model）

**锚点**是一切交互的起点。它必须是**有形的、可被体验的事物**，由此生发出来的才是真挚的感情。

```
                    真实的情感
                       ↑
               用户的反应/感受
                       ↑
    用户 A ──→  [ 锚 点 ]  ←── 用户 B
                    ↑
              现实中的某个事物
```

**没有锚点，就没有表达。** 这不是限制，是过滤器——它天然地过滤掉了"为了发而发"的内容。

#### 锚点类型

| 类型 | 示例 | 感受的来源 |
|---|---|---|
| 文字 | 书中的某段话、一篇深度报道 | "这句话击中了我" |
| 图片 | 一张街景照片、一幅画 | "这个画面让我想起……" |
| 视频 | 一段纪录片片段、一场演讲 | "这个人说的话我深有感触" |
| 声音 | 一首歌、一段环境音、一次口述 | "听到这个声音我就回到了那个瞬间" |
| 物件 | 一个杯子、一张旧车票 | "这个东西承载了一段记忆" |
| 场所 | 某个咖啡馆、某条街道 | "在这里我经历过……" |

#### 锚点来源（三种）

| 来源 | 形成方式 | 特点 |
|---|---|---|
| **平台初始锚点** | 编辑团队/AI 筛选高质量内容 | 冷启动用，保证基础质量 |
| **用户内容直接成为锚点** | 用户发布的观点/经历被系统识别为有锚点价值 | 原生、真实、多元 |
| **AI 聚合生成锚点** | 同主题下大量用户观点被归纳成完整的新锚点 | 代表群体共识，信息密度高 |

#### 锚点内容标准

- **完整性**：需要有充足的信息量和上下文，不是碎片化的标题
- **具体性**：有细节、有前因后果、有引发思考的空间
- **真实性**：基于真实存在的事物、事件、作品
- 参考标准：更像一篇深度报道的节选、一段纪录片的核心片段、一本书的核心章节

### 3.2 共鸣系统（Resonance System）

#### 五态反应

每条锚点内容和评论，用户都可以选择五种反应：

| 反应 | 含义 | 权重 |
|------|------|------|
| **共鸣** | "我也有同样的感受" | 强正向 |
| **无感** | "没什么特别的感觉" | 弱信号 |
| **反对** | "我不同意这个观点" | 强负向（也是一种深度参与） |
| **未体验** | "我认为 ta 没有体验过" | 真实性信号 |
| **有害** | "这是恶意的/虚假的/有害的" | 治理信号 |

用户可以在**任一反应状态**下留下自己的观点，进行观点碰撞。

**"共鸣"的情绪词细分**（可选）：共鸣也分很多种。点击"共鸣"后可选一个情绪词，精简且精准：

| 情绪词 | 含义 | 设计意图 |
|--------|------|---------|
| 同感 | "我也有同样的感受" | 最常见的共鸣类型 |
| 触发 | "让我想起了……" | 锚点触及了个人记忆 |
| 启发 | "没这么想过，但你说得对" | 视角互补 |
| 震撼 | "说不出话" | 被深深击中 |

情绪词可选——不选直接点"共鸣"也可以。精简 4 个覆盖共鸣的主要维度。

#### "未体验"标记的设计意图

不限制谁能发言，但通过社区标记来自然筛选不真实的分享。

- **软性约束**：不阻止任何人发表任何方向的观点，但允许其他用户标记"我认为 ta 没有体验过"
- **有成本**：和"有害"标记一样，需要写理由，不能只点按钮
- **影响权重**：被大量标记为"未体验"的内容，其"共鸣"权重会降低——不是删除，是降低影响力
- **不等于有害**："未体验"不是攻击，是对真实性的判断。大量标记为"未体验"的内容不会被屏蔽，但不会获得高共鸣权重
- **设计原则**：靠机制让真实的表达自然胜出，而非靠硬性限制谁能说

#### 内容治理机制

基于社区反应的内容治理（非平台审核员判定）：

| 触发条件 | 系统动作 |
|------|---------|
| 共鸣 vs 反对严重失衡 | 可能是争议性内容 → 降权但不屏蔽 |
| "有害"占比异常高 | 大概率恶意内容 → 屏蔽，停止扩散 |
| "未体验"占比高 | 内容真实性存疑 → 降低共鸣权重，不删除 |
| 每种状态下可留下观点 | 观点本身也受同样的治理规则约束 |

- "有害"不是"我不喜欢"，是有成本的信号，需要防止滥用
- "未体验"不是"我反对这个观点"，是对表达者真实性的判断
- 争议性内容（共鸣和反对都高）≠ 有害内容，前者保留但降权

#### 共鸣值计算逻辑

```
单锚点共鸣值 = f(反应类型, 观点深度, 相关性, 增量性, 有害惩罚, 未体验惩罚)
        │
        ↓
话题内衰减：同一话题第 N 次共鸣的边际价值递减
        │
        ↓
跨话题加权：不同话题/领域的共鸣 → 权重递增
        │
        ↓
关系分 = Σ(各锚点共鸣值 × 话题衰减) × 话题多样性加权
        │
        ↓
高阶权限触发 = 关系分 AND 活跃度信号（双方近期都在场）
        │
        ↓
达标 → 自然涌现沟通可能
```

**两层衰减 + 活跃度信号**：
- **话题内衰减**：同一话题第 N 次共鸣的边际价值递减——防止在一个话题上刷共鸣
- **跨话题加权**：不同话题/领域的共鸣权重递增——奖励世界观层面的深层连接
- **活跃度信号**：关系分不随时间衰减（共鸣是事实，不是状态），但高阶权限需要双方近期都有活跃——不是"共鸣过期了"，而是"对方不在了"

### 3.3 身份系统（Identity System）

#### 三层身份模型

```
注册阶段：强实名（手机号/真实身份验证，平台存档，永不对外暴露）
                │
                ↓
        ┌───────────────────────────────────┐
        │  一切对外展示                       │
        │  随机昵称                          │  ← 每次发言可能不同
        │  随机头像                          │  ← 系统分配
        │  无个人主页                        │  ← 无法主动查看某人历史发言
        └─────────────────┬─────────────────┘
                          │
                  共鸣积累 → 关系自然涌现
                          │
                          ↓
        ┌───────────────────────────────────┐
        │  感受知己                          │
        │  系统提示「有个人总是跟你有同样的感受」  │
        │  可见对方的「感受模式」              │
        │  不可见任何身份信息                  │
        └─────────────────┬─────────────────┘
                          │
                  双方独立表达「想认识这个人」
                          │
                          ↓
        ┌───────────────────────────────────┐
        │  匿名知己                          │
        │  互相可见「设定昵称」（非真名）       │
        │  互相可见「设定头像」（非真实照片）    │
        │  互相可见「设定简介」                │
        │  可自由对话                         │
        │  可随时退回「感受知己」或完全匿名     │
        └───────────────────────────────────┘
```

#### 核心原则：匿名性永不打破

- **强实名**：守住底线（违法、恶意行为可追溯），但永不对外暴露
- **展示全匿名**：消除社交表演压力，只看感受不看人
- **感受知己**：通过感受认识一个人，不是通过身份
- **匿名知己**：即使最深层的连接，也只展示用户自己设定的昵称和头像
- **真名永不暴露**：真名可以被搜索、可以跨平台追踪，一旦暴露无法收回
- **线下见面是独立功能**：不属于知己系统的自然延伸，需要双方单独选择交换真实身份

#### 为什么「设定昵称」而非「真名」？

即使在最深的知己关系中，匿名性也应该被保留：

- **真名可以被搜索** → 打破匿名性，可以在其他平台找到对方
- **真名可以跨平台追踪** → 一旦知道真名，无法「不知道」
- **真名是不可逆的** → 一旦暴露，无法收回
- **设定昵称是用户主动选择的** → 用户决定展示什么，保留控制权

知己关系的本质是「感受的连接」，不是「身份的连接」。
你通过感受认识一个人，不需要知道 ta 的真名。

## 四、交互模型

### 4.1 核心交互循环

```
平台/用户提供锚点（高质量、完整内容）
        │
        ↓
用户浏览、体验锚点
        │
        ↓
留下反应：共鸣 / 无感 / 反对 / 有害 ──→ 可在任一状态下留下观点
        │                                      │
        ↓                                      ↓
系统计算共鸣值（跨话题加权）          观点碰撞产生新的讨论
        │
        ↓
共鸣值达标 → 解锁匿名身份下的沟通渠道
        │
        ↓
持续共鸣积累 → 可选择互相揭示固定身份
        │
        ↓
多人共鸣达标 → 进入群体实时对话窗口
```

### 4.2 信任递进阶梯

体验是连续的渐进增强，而非离散的功能解锁。用户不会感觉到"我解锁了新东西"，只会感觉到"这个体验越来越丰富了"。

```
L0  浏览锚点 → 看到内容，可以留下反应
L1  痕迹可见 → 看到他人的匿名观点，体验观点碰撞
L2  观点回应 → 对他人观点做出回应，对方可能看到（异步）
L3  回达增强 → 回应时可选"确保对方看到"（仍异步，但有送达感）
L4  实时碰撞 → 双方同时在锚点下，观点回应变为实时
L5  群体共鸣 → 多人同时在锚点下深度交流
```

**关键：每一级都是前一级的"渐进增强"，而非"新增一个功能"。**
- L0→L1：不是"解锁了看别人的观点"，而是锚点下本来就有观点
- L2→L3：不是"解锁了私信"，而是回应方式多了一个选项
- L3→L4：不是"解锁了聊天"，而是同时在场时对话自然变实时
- 所有交流始终在锚点的上下文里，没有跳到独立的"聊天界面"

**阈值计算是双向的**——必须双方的共鸣关系分都达到阈值。系统不会告诉用户"你解锁了 Level X"——权限在后台静默开启，用户在自然体验中偶然发现。

#### 知己：感受的连接，不是身份的连接

##### 核心设计哲学

知己关系的本质是「通过感受认识一个人，不是通过身份」。

- **你不知道 ta 是谁**，但你知道 ta 总是跟你有同样的感受
- **你不需要知道 ta 的真名**，因为知己关系的价值在于感受的连接
- **你通过感受认识 ta**，不是通过名字、长相、职业

##### 关系深度是光谱，不是阶段

真实的关系不是二元的（连接/断开），而是连续的光谱（亲近/疏远）。

**系统内部维护一个「关系深度」光谱（0-100），但用户永远看不到这个数字。**

用户感知到的:
- 某个人的内容越来越多地出现在你的视野
- 某个人越来越频繁地与你共鸣
- 某天你突然发现：「有个人总是跟我有同样的感受」
- 你不知道「深度」是多少，你只是... 感觉到了

**关系光谱是系统内部的，不是用户界面的。**
**没有「共鸣深度」数字，没有「关系进度条」，没有任何指标。**

##### 升级：自然发生，不是手动操作

知己关系的升级是**自动的、基于共鸣的、无声的**：

```
完全匿名（初始状态）
    │
    │  你们在多个锚点上反复共鸣
    │  系统检测到共鸣模式相似
    │  一切自动发生，你没有「选择」任何东西
    ↓
感受知己（系统自动识别）
    │
    │  系统提示：「有个人总是跟你有同样的感受」
    │  你可以看到对方的「感受模式」（匿名的）
    │  不能看到任何身份信息
    │
    │  如果双方都独立选择「想认识这个人」
    ↓
匿名知己（双向选择）
    │
    │  互相可见「设定昵称」「设定头像」「设定简介」
    │  可自由对话
    │  仍然不知道对方的真名
    │
    │  如果双方都选择交换真实身份（独立功能）
    ↓
线下见面（独立功能，不属于知己系统）
```

**关键原则**：
- 你没有「找知己」，你只是表达自己，然后遇到了一个跟你很像的人
- 升级是自动的，不是你选择的
- 只有「匿名知己」需要双方主动选择，其他升级都是自然发生的
- 线下见面是独立功能，不属于知己系统的自然延伸

##### 降级：渐进的，不是突然的

知己关系的降级是**无声的、渐进的、没有仪式感的**：

- **不是**：「断开连接」按钮（突然的，有情感成本）
- **而是**：减少互动，关系自然淡化

你不需要「做」什么来降级，你只需要「不做」什么。

- 你只是不再跟这个人共鸣
- 系统自然地减少这个人的曝光
- 关系自然地淡化
- 没有「断开」的仪式感，没有情感成本
- 对方不会收到任何通知

**降级是无声的，对方不会知道。**
**没有「删除好友」「断开连接」的操作，只有自然的疏远。**

##### 关系容量与静默机制

**核心洞察**：每个人的精力是有限的。你可能跟 100 个人有共鸣，但你最多只能跟 5-10 个人建立深度连接。

**关系深度不只是绝对值，也是相对排名。**

```
你的所有共鸣关系
    │
    │  按关系深度排名
    ↓
┌─────────────────────────────────────┐
│  前 N%：显著曝光                      │
│  在「感受知己」区域突出展示             │
│  N 基于你的互动容量，因人而异           │
└─────────────────────────────────────┘
    │
    │  排名下降
    ↓
┌─────────────────────────────────────┐
│  其他：静默                           │
│  仍然存在，但降低能见度                │
│  不是删除，不是惩罚                   │
│  只是你的注意力自然流向最强的连接       │
└─────────────────────────────────────┘
```

**静默机制的设计原则**：
- **静默不是删除**：关系仍然存在，只是降低能见度
- **静默是无声的**：对方不会知道自己被静默了
- **静默是可逆的**：如果你们再次共鸣，排名可能上升，关系恢复
- **静默是自然的**：不需要手动操作，系统自动处理
- **容量是个性化的**：基于你的互动频率，不是固定数字

**静默是关于你的注意力，不是关于关系本身。**
你有有限的精力，系统帮你把注意力分配给最强的连接。

##### 知己入口的设计原则

**功能渐进解锁**：知己功能不对新用户展示。只有当用户的使用达到可以建立知己关系的程度时，相应的功能窗口才会自然出现。

```
新用户（初始状态）
    │  看不到任何知己相关的 UI
    │  不知道有知己功能的存在
    │  专注于表达和共鸣本身
    ↓
共鸣积累到阈值
    │  系统检测到你已经有足够深的共鸣关系
    │  知己入口自然出现（安静地，不通知）
    │  你发现它时，里面已经有内容了
    ↓
建立匿名知己
    │  双方选择「想认识这个人」
    │  匿名知己功能解锁
    ↓
知己关系足够深
    │  系统检测到关系足够稳定
    │  线下见面选项自然出现（安静地）
    │  你需要它时，它已经在了
```

**设计原则**：
- 不在底部 Tab 栏（不是一等公民）
- 不在首页突出展示
- 入口位置：「我」页面内的一个安静条目
- 新用户看不到这个入口（功能未解锁）
- 当你第一次看到它时，里面已经有内容了——那一刻的惊喜感：「原来这里有人了」
- 没有数字角标，没有引导，没有「去发现知己」的提示

##### 线下见面：同样是渐进解锁

线下见面功能不随匿名知己一起解锁。只有当知己关系足够深、足够稳定时，系统才会安静地展示这个选项。

- 不是「解锁了线下见面」
- 而是「你们的关系已经深到可以考虑线下见面了」
- 选项出现时，你们已经建立了足够的情感基础
- 你不需要它时，它不会打扰你
- 你需要它时，它已经在了

##### 知己不违背「有物才有真实感想」

锚点内交流是关系的起点，知己自由沟通是关系已经足够深之后的自然延伸。他们对话的根基是之前在无数锚点上积累的共鸣。

### 4.3 沟通规则

- **异步为主，体验自足**：主体是时间胶囊系统，异步体验本身必须足够好——留下感想不期待被谁看到，表达本身就是价值。核心自足体验包括：

  **① 观点碰撞**：即使不知道对方是谁，锚点下的不同观点之间的碰撞本身就有价值。用户按共鸣/反对/无感分组看到不同感受的人如何思考同一个问题——这种"思想的交锋"不需要建立人际关系。

  **② 共鸣预感**：用户留下反应后，系统显示"有 X 人和你感受相同"（只显示数字，不显示身份）。让用户知道"你不是一个人在想这件事"——即使共鸣关系尚未建立，被理解的感觉已经开始产生。

  **③ 群体记忆重现**：当锚点随时间回归（季节性、周年、社会事件回响），用户重新遇到同一个锚点时，系统浮现"群体记忆"——你一年前在这个锚点下的感想、当时有多少人和你感受相同、群体对这个话题的理解如何演变。**锚点是记忆的触发器，没有锚点的空泛时间胶囊不符合 Standby 的产品哲学（有物才有真实感想）。**

  **④ 表达即价值**：用户写下的观点本身就是对自我思考的整理。产品叙事和 UI 设计应传递"写下来就是收获"——而非"写下来等别人看到"。

- **实时是自然涌现，不是目标**：不告诉用户"你可以解锁聊天"，不存在进度条或成就系统，不让用户产生"还差多少就能聊天"的目标感
- **发现是一种惊喜**：用户在自然浏览中反复遇到同一个人的感想，逐渐意识到"有一个人一直在同一个频率上"——这种发现的惊喜感本身就是体验的一部分
- **不允许主动寻找目标**：用户不能搜索特定用户发起对话
- **实时对话的浮现方式**：当双方共鸣积累到足够程度且同时在线、浏览同一个锚点时，对话窗口自然浮现——"这里刚好有人在"，而非"你被邀请进入聊天"
- **即使永远没有发现对方，体验也是完整的**：自我表达、观点碰撞、群体记忆、共鸣预感——这些不依赖外部连接

### 4.4 跨设备体验

**目标生态**：对小米全生态多终端统一系统（澎湃 OS）进行原生级别适配，实现多端查看、多端交互、多端分享。

**一期覆盖设备**：

| 设备 | 场景定位 | 核心交互方式 |
|------|---------|------------|
| 手机 | 碎片时间、移动场景 | 浏览锚点、快速反应、轻量表达、多模态捕捉 |
| 平板 | 放松状态、深度场景 | 深度阅读、长文创作、沉浸浏览 |
| 电脑 | 书桌前、专注场景 | 长篇感想、系统性回顾、整理思考 |
| 车机 | 驾驶场景 | 语音收听、语音触发感想、情境推送 |

**跨设备的两层价值**：

#### 第一层：场景接力

用户的思考是连续的，但使用设备的场景是断续的。跨设备的核心价值之一是在场景切换时不让体验断裂。

典型接力场景：
- **触发 → 沉淀**：早晨通勤（手机）看到锚点被击中，留下简短反应；晚上（平板）想法发酵后写下完整感想
- **收听 → 思考**：开车（车机）听到锚点语音摘要深受触动；到达后（手机）用语音记下感受；回到家（电脑）整理成完整观点
- **浏览 → 分享**：在网页上看到深度文章（电脑），通过系统级分享直接导入成为锚点素材（手机）
- **碎片发现 → 完整回顾**：手机上偶然发现熟悉风格的感想，电脑上在共鸣记录里拼出完整的共鸣轨迹

#### 第二层：多设备融合感知（核心差异化价值）

多设备并行融合能产生单一设备永远无法产生的状态理解，从而提供单一设备永远无法提供的体验质量。

**多维度状态理解**：

单一设备只能看到用户的一个切面——手机知道你在哪但不知道你在想什么，平板知道你在看什么但不知道你的情绪状态。但当多设备信号融合时：

- 手机感知你在咖啡馆 + 平板感知你正在深度浏览关于孤独的锚点 + PC 感知你已连续工作 4 小时 → 系统理解"一个疲惫的人在安静的咖啡馆里被一段关于孤独的话击中了"
- 车机感知你在深夜独自开车 + 手机感知你在听一首老歌 + 平板感知你昨天写过关于回忆的感想 → 系统理解"这个人正处于怀旧的情绪状态中"

**单一设备永远无法得出这种判断。这是融合的真正价值。**

**锚点的情境增强**：

同一个锚点在不同情境下对人的触动完全不同。多设备融合让系统理解的不只是"用户看了什么"，而是"用户在什么状态下看了什么"。锚点的呈现更有温度——在对的时间、对的状态下，把对的内容送到面前。

**表达的多模态融合**：

人的感受本身就是多模态的。多设备融合可以同时捕捉：手机摄像头的画面 + 耳机的音乐 + 语音记录的感受，合成为一个完整的锚点素材。这个表达质量是任何单一设备都做不到的。

**共鸣的空间感知**：

当多设备感知到两个用户在物理空间上接近且浏览同一个锚点时，这种"同空间 + 同锚点"的信号可以在共鸣机制中加权。不需要暴露位置——只是两个在同一个咖啡馆被同一段话击中的人，他们的共鸣可能比纯线上更真实。

**生活节奏同步**：

多设备长时间融合感知能拼出用户的日常生活节奏——每晚 10 点后是深度阅读时间（平板）、每次通勤路上倾向快速浏览（手机）、每次开车时情绪最放松（车机）。系统在合适的节奏点上提供合适的体验，不是"产品在服务你"，而是"产品在和你同步呼吸"。

#### 与小米智能助理的深度融合

不只是通信接口，智能助理是跨设备体验的调度者：
- **情境感知**：知道用户在哪个设备上、什么状态，自动调整交互方式
- **主动推送**：根据用户习惯，在合适的设备上、合适的时间，轻推可能引发共鸣的锚点
- **语音桥接**：用户说"帮我记一下"，智能助理自动关联上下文（正在听的歌、正在看的视频、正在经过的地方）
- **设备接力**：在手机上浏览到一半，智能助理建议"这个内容很适合在平板上细读"——不强制，只是轻提示

#### 隐私红线

所有融合感知必须遵循：
1. **所有原始数据不离开设备**——融合在端侧进行，只输出抽象状态，不输出原始传感器数据
2. **用户完全知情和可控**——用户可以查看"系统感知到了什么"，可以关闭任意维度的感知
3. **不对外暴露任何感知信息**——其他用户、合作方、任何第三方都无法获取
4. **感知服务于体验，不服务于商业**——不基于融合感知做广告、不做行为画像变现

**设计原则**：跨设备不是功能叠加，是让产品融入用户在不同设备间流转的日常生活。多设备融合的真正价值不是"无缝切换"，而是产生单一设备永远不可能产生的状态理解，提供单一设备永远不可能提供的体验质量。

## 五、AI 角色定义

### 5.1 核心技术资产：五个 AI 决策引擎

Standby 的核心技术资产是五个 AI 驱动的决策引擎，它们协同工作，构成产品的技术护城河。

```
                    ┌─────────────────────┐
                    │   情境感知引擎       │
                    │ （多设备融合感知）    │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              ↓                ↓                ↓
     锚点生成引擎        共鸣机制引擎      内容治理引擎
     （情境增强锚点）   （空间感知加权）   （情境辅助判断）
              │                │
              ↓                ↓
          用户管理引擎
       （节奏适配交互）
```

| 引擎 | 核心职责 |
|------|---------|
| 情境感知引擎 | 理解用户状态和情境（多设备融合感知） |
| 锚点生成引擎 | 产出高质量内容 |
| 共鸣机制引擎 | 计算人与人之间的共鸣 |
| 用户管理引擎 | 管理身份与权限 |
| 内容治理引擎 | 保障社区安全 |

情境感知引擎是底层感知层——其他四个引擎都从它获取情境信息，但它不直接参与内容、关系、身份或安全的决策。

**部署原则：独立部署、可解耦单独扩容**

五个引擎独立部署，通过标准化接口通信，任何一个引擎可以单独扩容而不影响其他引擎。各引擎根据负载特征采用不同的扩容策略：

| 引擎 | 扩容特征 |
|------|---------|
| 情境感知引擎 | 端侧运行，无需云端扩容 |
| 锚点生成引擎 | 离线批处理，按日均产出量扩容 |
| 共鸣机制引擎 | 增量计算，按用户活跃度扩容 |
| 用户管理引擎 | 轻量计算，按用户总量扩容 |
| 内容治理引擎 | 按内容标记频率扩容，可用端侧初步过滤降低云端负载 |

AI 引擎成本可控——端侧优先、离线批处理、增量计算、模型分级。整体运营成本不高于传统社交平台（无需养推荐算法团队、无需海量人工审核、带宽成本低）。

### 5.2 锚点生成引擎

从多种来源提炼出符合锚点标准（完整、有信息量、能引发深层共鸣）的高质量内容。

**能力要求**：
- **深度分析**：对原始素材进行深入理解和重构，不是简单摘要或搬运
- **标准把控**：确保输出的锚点内容完整、具体、真实，有充足的信息量和上下文
- **来源多样**：支持从经典文学、纪录片、深度报道、历史事件、哲学论述、音乐等多种来源提炼
- **分歧保留**：当归纳群体观点时，忠实反映共识和张力，不制造"中位数观点"
- **来源标注**：标注锚点的来源和提炼方式，保持透明

**关键约束**：
- 不做简单或粗糙的内容整合
- 每个锚点都需要经过深度分析，保证信息密度和引发思考的空间
- 输出必须经得起时间检验——三个月后读到仍然能引发共鸣

### 5.3 情境感知引擎

底层感知层。融合多设备信号，理解用户的状态和情境，为其他引擎提供情境信息。

**核心能力**：
- **多设备信号采集**：从手机、平板、电脑、车机收集传感器和行为信号（位置、运动、使用状态、时间模式等）
- **端侧融合**：所有原始数据不离开设备，在端侧进行融合计算，只输出抽象情境状态
- **多维度状态理解**：融合多个设备的信号，推断用户的情绪状态、注意力状态、场景类型（如"疲惫的人在咖啡馆被一段关于孤独的话击中"）
- **生活节奏建模**：长时间学习用户的设备使用节奏和情感周期，在合适的节奏点上提供合适的内容和交互
- **空间感知**：感知用户间的物理空间关系（不暴露位置，仅用于共鸣加权等内部计算）

**与其他引擎的协作**：
- 给锚点生成引擎：提供情境信息，增强锚点的呈现时机和方式
- 给共鸣机制引擎：提供空间感知信号，用于共鸣关系加权
- 给用户管理引擎：提供用户节奏信息，适配不同设备上的交互方式
- 给内容治理引擎：提供情境信息，辅助判断内容的上下文

**关键约束**：
- 原始数据不离开设备，只输出抽象状态
- 用户完全知情和可控——可查看感知内容、可关闭任意维度
- 不对外暴露——其他用户、合作方、第三方均无法获取
- 感知服务于体验，不服务于商业——不做广告、不做行为画像变现

### 5.4 共鸣机制引擎

发现人与人之间隐藏的共鸣，计算共鸣关系，驱动连接的自然涌现。

**核心能力**：
- **五态反应处理**：对每条锚点内容和评论，处理用户的共鸣/无感/反对/未体验/有害五种反应
- **共鸣值计算**：基于反应类型、观点深度，计算单锚点共鸣值
- **话题内衰减**：同一话题第 N 次共鸣的边际价值递减
- **跨话题加权**：不同话题/领域的共鸣权重递增，奖励世界观层面的深层连接
- **活跃度信号**：关系分不衰减，但高阶权限需要双方近期都有活跃
- **关系分计算**：综合各锚点共鸣值与话题多样性系数

#### 关系深度算法

##### 核心原则

1. **用相对值，不用绝对值**：关系深度 = 你们的共鸣频率 / 你的正常共鸣频率
2. **关系深度对用户不可见**：系统内部维护，用户永远看不到任何数字
3. **个性化基线**：每个用户有自己的互动节奏，衰减速率因人而异
4. **跨时空友好**：关系可以「暂停」而非「衰减」，长时间沉默不会惩罚关系
5. **容量限制**：前 N% 关系获得显著曝光，其他关系静默

##### 个性化基线

每个用户有一个「互动基线」，动态更新：

- **表达频率**：你多久表达一次？
- **共鸣频率**：你多久共鸣一次？
- **共鸣比例**：你看到的内容中，多少比例会共鸣？
- **平均互动间隔**：你的正常互动节奏是什么？

**基线是你自己的「正常水平」，不是全局标准。**

##### 相对共鸣强度

对于每对用户 (A, B)，计算「相对共鸣强度」：

```
相对共鸣强度 = 
  (A 和 B 的共鸣次数) / 
  (A 的正常共鸣频率 × B 的正常共鸣频率 × 时间窗口)
```

**举例**：
- A 每天表达，正常共鸣率 20%
- B 每周表达，正常共鸣率 50%
- 过去一个月，A 和 B 共鸣了 10 次
- 对 A：正常共鸣次数 = 30天 × 1次/天 × 20% = 6次 → 相对强度 = 10/6 = 1.67
- 对 B：正常共鸣次数 = 4周 × 1次/周 × 50% = 2次 → 相对强度 = 10/2 = 5.0

**同样的 10 次共鸣，对 B 来说更「特殊」。**

##### 个性化衰减

衰减不是基于绝对时间，而是基于「你的互动节奏」：

```
衰减条件：距离上次互动 > N × 你的平均互动间隔
衰减速率：基于你的「沉默-回归」模式
```

- **N 的取值是个性化的**：基于用户的历史行为
- **沉默容忍度**：如果用户经常沉默但总会回来，降低衰减速率
- **沉默含义学习**：系统学习每个用户的「沉默含义」，个性化处理

##### 关系暂停机制

对于跨时空互动的关系，不使用「衰减」，而使用「暂停」：

```
当双方沉默时：关系分数「冻结」
当双方再次互动时：关系分数「恢复」
```

**长时间沉默不会「惩罚」关系，只是「暂停」。**
**这对于跨时空互动非常重要。**

##### 性能优化：事件记录与关系计算分离

```
事件记录（实时，轻量）
    │  每次共鸣，只记录一个事件
    │  O(1) 操作，非常轻量
    ↓
关系计算（批量，异步）
    │  每天凌晨，批量计算所有关系深度
    │  只计算「有新事件」的配对
    │  计算结果缓存起来
    ↓
用户查询（缓存，即时）
    │  用户打开 app，查询缓存的关系分数
    │  O(1) 操作，非常快
```

**可选优化**：
- **增量更新**：不重新计算整个关系分数，只计算「这次共鸣带来的增量」
- **懒加载**：只计算「用户打开 app 时需要的」关系分数，不计算不活跃用户

##### 关系容量与静默

```
计算所有关系的深度
    │
    │  按深度排名
    ↓
前 N%：显著曝光（在「感受知己」区域突出展示）
    │
    │  N 基于你的互动容量，因人而异
    │  容量 = 你能维持的深度关系数量
    │  基于你的互动频率动态调整
    ↓
其他：静默（存在但降低能见度）
```

**静默不是删除，是注意力分配。**
**被静默的关系可以随时恢复（再次共鸣即可）。**

**关键约束**：
- 共鸣值不公开显示，没有排行榜
- 不做"你可能认识的人"推荐
- 共鸣痕迹的呈现遵循"自然涌现"原则——不主动推送，让用户自行发现
- 关系深度对用户完全不可见，没有进度条、没有数字、没有排名

### 5.5 用户管理引擎

管理匿名身份、权限递进、信任关系。

**核心能力**：
- **双层身份管理**：强实名注册存档 + 全匿名对外展示
- **信任递进**：基于共鸣值静默开启各级权限（Level 0-5），不通知用户"已解锁"
- **身份揭示**：当双方持续共鸣积累到足够程度时，允许双方选择互相揭示固定身份
- **信用记录**：维护标记者的历史准确率等信用数据，供内容治理引擎查询

### 5.6 内容治理引擎

实时监控社区安全，防止有害内容和滥用行为，同时保护真实表达。

**核心能力**：
- **有害内容检测**：识别虚假信息、恶意攻击、垃圾营销等有害内容
- **异常模式识别**：检测协同攻击、刷量、有组织标记等异常行为
- **标记者信用评估**：基于历史准确率计算标记者权重
- **共鸣/有害冲突判定**：当内容同时获得大量共鸣和大量有害标记时，触发分歧标注而非直接删除
- **分级响应决策**：根据检测结果决策内容的可见度（观察→降权→暂停→移除）

**与其他引擎的协作**：
- 接入锚点生成引擎：检测锚点内容本身是否有害
- 接入共鸣机制引擎：监控共鸣/有害模式异常
- 接入用户管理引擎：查询标记者信用

**关键约束**：
- 不做价值判断，只做异常检测
- AI 输出为辅助信号，非最终决定
- 争议性内容（共鸣和反对都高）≠ 有害内容，前者保留但降权

### 5.7 AI 的边界

AI **不**做的事：
- 不生成用户观点（用户的一切表达必须来自用户自己）
- 不推荐用户（不基于画像做"你可能认识的人"）
- 不优化传播（不做"让更多人看到"的算法）
- 不评判对错（不判定某个观点是否"正确"）

## 六、产品与现有产品的本质区别

| 维度 | 微博/小红书 | 微信朋友圈 | Soul/陌陌 | 日记 App | **Standby** |
|---|---|---|---|---|---|
| 驱动力 | "我想让别人看到我" | "我的熟人在看" | "我想认识人" | "我想记录自己" | **"这个事物让我有话想说"** |
| 内容逻辑 | 流量驱动 | 关系链驱动 | 匹配驱动 | 纯私人 | **共鸣驱动** |
| 关系建立 | 先关注人 → 看内容 | 已有关系 | 算法匹配 | 无社交 | **先共享体验 → 发现人** |
|| AI 角色 | 推荐算法 | 无 | 匹配算法 | 无 | **五引擎协同** |
| 真实性 | 低（流量表演） | 中（熟人表演） | 低（匿名随意） | 高（纯私人） | **高（匿名但真诚）** |
| 身份机制 | 半实名 | 强实名 | 匿名 | 无 | **强实名注册 + 全匿名展示** |

### Standby 的一句话定位

**别的产品是"人→内容→人"，Standby 是"人→事物→感受→人"。中间多了一个"事物"锚点，一切都不一样了。**

## 七、AI 时代的核心洞察

### 为什么现在需要 Standby？

AI 时代有一个悖论：

- AI 让**生成内容**变得极度容易 → 内容泛滥，真假难辨
- AI 让**理解内容**变得极度容易 → 但没有人用它来做这件事

大多数产品选择用 AI 来**生成更多内容**（AI 写作、AI 绘画、AI 视频），进一步加剧了内容通胀。

Standby 选择用 AI 来**理解已有内容**：
- 理解用户表达中的真实意图
- 发现人与人之间隐藏的共鸣
- 将碎片化的个体声音编织成完整的图景

**AI 不应该是人与人之间的隔阂，而应该是桥梁。**

## 八、商业模式

### 8.1 核心立场

Standby 不采用以下商业模式：
- **不卖用户数据**：用户的一切表达、反应、共鸣关系不对外提供
- **不做广告**：不以流量变现，不存在广告推荐算法
- **不搞会员分级**：不通过付费解锁功能来区分用户
- **不做企业版定制**：不为 B 端客户改变产品对用户的设计原则

### 8.2 变现路径：跨设备生态合作 + 咨询服务

#### 路径一：跨设备生态合作

作为跨设备智能交互能力的提供者，嵌入设备厂商生态：

- **SDK/中间件**：将核心能力封装为 SDK，设备厂商集成到自有系统中，按设备出货量或激活量收取授权费
- **深度合作/联合开发**：与特定生态厂商绑定，成为其跨设备 AI 体验的核心组件
- **标准/协议参与**：推动跨设备智能交互的行业标准，提供参考实现

**目标生态**：小米全生态多终端统一系统（澎湃 OS）。

**合作形态**：
- **原生级别适配**：对澎湃 OS 进行系统级适配，实现手机、平板、电脑、车机四端的多端查看、多端交互、多端分享
- **智能助理通信**：与小米系统级智能助理（小爱同学）双向通信，遵循小米多终端智能助理开放协议
- **SDK/中间件**：将核心能力封装为 SDK，按设备出货量或激活量收取授权费
- **深度合作/联合开发**：成为小米跨设备 AI 体验的核心组件

**远期扩展**：在小米生态验证成熟后，可考虑向其他有自研 OS 的厂商扩展。

#### 路径二：基于用户洞察的咨询服务

Standby 的共鸣系统和 AI 织网者天然积累了对群体情感趋势、文化议题、社会心态的深度理解。这些洞察以完全匿名、聚合的方式提炼，不涉及任何个人数据。

**服务形态**：
- **趋势洞察报告**：基于匿名聚合数据，提供特定领域（文化、消费、社会心态）的趋势分析
- **议题研究**：针对特定社会议题，呈现公众的共鸣分布、观点分歧和演化趋势
- **决策咨询**：为政策制定者、学术机构、文化机构提供基于真实公众情感的数据支撑

**隐私保障机制**：
- 所有洞察基于匿名聚合，单条数据不输出
- 最小聚合门槛：只有参与人数达到阈值的数据才可提炼
- 差分隐私：在聚合结果中加入噪声，防止通过交叉比对反推个体
- 不提供用户画像、不提供行为追踪、不提供精准定位
- 洞察输出仅保留趋势和分布，不保留原始表达

**设计原则**：咨询的价值来自对群体情感的真实理解，而非对个体行为的追踪。Standby 提供的是\"这个时代人们在想什么\"，而非\"某个人在做什么\"。

## 九、\"有害\"信号防滥用机制

### 9.1 核心矛盾

\"有害\"信号需要够灵敏以识别真正有害的内容，同时够防弹以防止成为压制异见的武器。最危险的不是攻击本身，而是用户感知到风险后的寒蝉效应——与 Standby 保护真实表达的核心理念直接冲突。

### 9.2 可能的滥用形态

1. 协同攻击：一群人同时标记某条内容为有害，利用人数压制
2. 渐进渗透：少量账号积累信用后集中使用
3. 反向博弈：用大量\"共鸣\"推高有害内容，再用\"有害\"标记打击正常内容，制造混乱
4. 寒蝉效应：用户感知到\"有害\"标记容易被用，不敢表达争议性观点

### 9.3 设计方案

#### 标记有成本

- **必须写理由**：不能只点按钮，必须输入说明为什么认为有害（最低字数要求）
- **标记者可审计**：对外匿名，但系统记录标记者的实名身份，恶意标记可追责

#### AI 前置分析，不做最终裁决

AI 守护者在收到\"有害\"标记后进行分析：
- 内容本身是否符合有害特征（虚假信息、恶意攻击、垃圾营销等）
- 标记模式是否异常（短时间内大量标记、标记者之间存在关联）
- 标记者的历史准确率

AI 输出为辅助信号，非最终决定。

#### 分级响应，而非二元处理

| 阶段 | 触发条件 | 系统动作 |
|------|---------|---------|
| 观察 | 少量有害标记 | 不做任何处理，记录 |
| 降权 | 有害标记达到阈值 + AI 初步确认 | 降低可见度，但不屏蔽 |
| 暂停 | 有害标记继续增加 + AI 高置信度 | 暂停传播，进入复核 |
| 移除 | 复核确认有害 | 移除，通知标记者结果 |
| 反转 | 复核确认无害 | 恢复，消耗标记者信用 |

在\"降权\"阶段，内容仍然可见，只是不再被主动推送，给系统留缓冲空间。

#### 标记者信用体系

- **历史准确率**：过去标记被复核为正确的比例越高，权重越大
- **反噬机制**：标记被复核为\"无害\"，标记者信用下降，未来标记权重降低
- **严重滥用惩罚**：多次恶意标记 → 降低标记权重 → 限制标记权限
- 信用体系不对外暴露，避免博弈

#### 共鸣与有害的冲突检测

如果一条内容同时获得大量\"共鸣\"和大量\"有害\"标记：
- 不直接删除
- 触发分歧标注：系统自动标注\"此内容引发较大争议，有 X% 的人共鸣，Y% 的人认为有害\"
- 让用户自行判断

争议 ≠ 有害。系统不代替用户做判断。

#### 申诉和纠错机制

- 被标记为有害的内容创作者有权知道自己的内容被标记（但不知道是谁标记的）
- 可提交申诉
- 申诉结果公开透明

### 9.4 实施优先级

- P0（一期必须有）：标记必须写理由、分级响应、共鸣/有害冲突检测
- P1（尽快做）：标记者信用体系、AI 异常检测
- P2（可以后做）：申诉机制

## 十、成功指标

### 核心原则

留存率、注册量、使用时长是**结果**，不是目标。追逐这些指标会导致讨好用户的设计，背离"真实表达"的产品哲学。真正需要追踪的是**核心机制是否在运转**。

### 指标分层

#### 第一层：核心行为（机制是否运转）

| 指标 | 含义 | 追踪方式 |
|------|------|---------|
| 反应率 | 看锚点后留下反应的比例 | 观测，期望 > 40% |
| 观点率 | 留反应后附带文字的比例 | 观测，期望 > 20% |
| 情绪词选择率 | 共鸣时选择情绪词 vs 纯点击 | 观测，验证情绪词设计 |
| 观点碰撞参与率 | 看到他人观点后留下回应的比例 | 观测 |

#### 第二层：核心价值（共鸣是否发生）

| 指标 | 含义 | 追踪方式 |
|------|------|---------|
| 群体共鸣预感触发展示率 | 多少用户看到了"有人和你一样" | 观测 |
| L3+ 关系达成率 | 多少用户对达到 L3 以上 | 一期追踪，不设目标 |
| 知己建立率 | 知己关系的建立数量和趋势 | 一期追踪，不设目标 |
| 首次共鸣发现天数 | 从首次反应到发现共鸣痕迹的平均天数 | 观测 |

#### 第三层：体验自足（底座是否稳固）

| 指标 | 含义 | 追踪方式 |
|------|------|---------|
| 无共鸣用户 D7 留存 | 没有共鸣关系的用户是否仍然留下来 | 期望 > 25%，验证自足体验 |
| 群体记忆参与率 | 用户在锚点重现时的互动比例 | 观测 |
| 表达自足率 | 不期待被看到但仍留下观点的用户比例 | 观测 |

#### 第四层：成功验证（飞轮是否转起来）—— 核心中的核心

| 指标 | 含义 | 追踪方式 |
|------|------|---------|
| 用户锚点通过率 | 用户内容被识别为有锚点价值的比例 | 观测趋势 |
| 用户锚点表现对比 | 用户锚点 vs 平台锚点的反应数/共鸣率/观点率 | **无显著差异 = 飞轮转了** |
| 用户锚点占比趋势 | 用户自产锚点在总锚点中的月度占比 | 观测上升趋势 |
| 用户锚点时间检验 | 用户自产锚点的群体记忆触发率 | 长期追踪 |

**最终检验标准**：当用户自产的锚点和平台初始锚点在各项表现指标上没有显著差异时——产品成功了。用户不再只是消费者，而是和平台一起创造高质量内容。

### 结果指标（仅观测，不追逐）

| 指标 | 说明 |
|------|------|
| D1/D7/D30 留存 | 机制运转的结果，不是目标 |
| 注册量 | 产品的自然结果 |
| 日均使用时长 | 不追求最大化，体验自足即可 |

| 11 | 商业模式：跨设备生态合作 + 咨询服务 | 不卖数据、不做广告、不搞会员分级，收入来自厂商授权和匿名聚合洞察 |

| 13 | 分级响应而非二元删除 | 降权→暂停→移除逐级处理，给系统留缓冲空间 |
| 14 | 共鸣/有害冲突检测 | 争议 ≠ 有害，系统不代替用户做判断，触发分歧标注 |
| 15 | 标记者信用体系不对外暴露 | 防止用户博弈信用分数 |

| 17 | 跨设备首选小米澎湃 OS 生态 | 原生级别适配手机/平板/电脑/车机四端，与小爱同学双向通信；小米生态验证成熟后再考虑扩展 |
| 18 | 核心技术资产为五个 AI 决策引擎 | 情境感知、锚点生成、共鸣机制、用户管理、内容治理五引擎协同，构成技术护城河 |
| 19 | 锚点生成引擎必须深度分析 | 不做简单搬运或粗糙整合，每个锚点经过深入理解和重构，保证信息密度和持久共鸣价值 |
| 20 | 跨设备体验分两层：场景接力 + 融合感知 | 场景接力解决体验断裂，融合感知产生单一设备无法提供的状态理解和体验质量；原始数据不离开设备 |
| 21 | 反应从四态扩展为五态，新增"未体验" | 不硬性限制发言资格，通过社区标记自然筛选不真实分享；"未体验"有成本、影响共鸣权重但不删除内容 |
| 22 | 五个引擎独立部署、可解耦单独扩容 | 标准化接口通信，各引擎按负载特征独立扩容，端侧优先控制成本 |
| 23 | 知己关系是「感受知己」，不是「身份知己」 | 即使最深层连接也只展示用户设定的昵称/头像，真名永不暴露；知己通过感受连接，不是通过身份 |
| 24 | 关系深度是光谱，不是阶段 | 系统内部维护 0-100 的关系深度光谱，但用户永远看不到；没有「升级按钮」「降级滑块」，只有自然的流动 |
| 25 | 升级/降级是自然发生的 | 升级基于共鸣积累自动发生，降级基于减少互动自然淡化；没有「断开连接」的操作，没有情感成本 |
| 26 | 个性化基线 + 相对共鸣强度 | 每个用户有自己的互动基线，关系深度基于相对共鸣强度（相对于你的正常水平），不是绝对值 |
| 27 | 关系暂停而非衰减 | 跨时空互动的关系在沉默时「冻结」分数，再次互动时「恢复」；长时间沉默不惩罚关系 |
| 28 | 关系容量与静默机制 | 每个人精力有限，前 N% 关系获得显著曝光，其他关系静默（降低能见度但不删除）；静默是无声的、可逆的、个性化的 |
| 29 | 性能优化：事件记录与关系计算分离 | 共鸣事件实时记录（O(1)），关系深度批量计算（每天），用户查询走缓存（O(1)） |
| 30 | 线下见面是独立功能 | 不属于知己系统的自然延伸，需要双方单独选择交换真实身份 |
| 31 | 功能渐进解锁 | 知己功能不对新用户展示，共鸣积累到阈值后自然出现；线下见面同理，关系足够深时才展示选项；用户不需要时它不打扰，需要时它已在 |
| 32 | 知己入口在「我」页面内 | 知己不是独立 Tab，入口在「我」页面内的一个安静条目 |
