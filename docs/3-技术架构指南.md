# Standby — 技术架构指南

> 产品理念驱动的技术选型。所有技术决策以产品需求为硬约束。

---

## 一、技术选型总纲

### 1.1 选型原则

| 原则 | 含义 |
|------|------|
| **隐私安全第一** | 每一层、每一个组件都必须贯彻数据安全与隐私保护 |
| **最低开销** | 优先选择资源占用低的方案，开发阶段可在单机运行 |
| **端侧优先** | 能在设备上跑的计算不下放到云端 |
| **AI 高兼容** | 对 AI 框架有极高的兼容性，支持端侧推理和云端 API |
| **独立部署** | 引擎间解耦，可单独扩容 |

### 1.2 技术栈全景

```
┌─────────────────────────────────────────────────────────────┐
│                       客户端                                  │
│  Flutter (Dart) ←→ flutter_rust_bridge ←→ Rust 核心层       │
│  UI渲染/交互     FFI桥接                 端侧推理/安全/加密    │
├─────────────────────────────────────────────────────────────┤
│                       后端服务                                │
│  Rust（基础服务、安全组件、引擎编排）                           │
│  Python（AI 引擎：心物生成、共鸣计算、内容治理）                 │
│  gRPC（引擎间同步通信）+ NATS（引擎间异步消息）                  │
├─────────────────────────────────────────────────────────────┤
│                       数据层                                  │
│  PostgreSQL（全量数据）         │
│  SQLCipher（端侧加密存储）+ Dragonfly（缓存）                  │
├─────────────────────────────────────────────────────────────┤
│                       安全基础设施层                           │
│  独立的安全组件贯穿所有层                                      │
└─────────────────────────────────────────────────────────────┘
```

| 层 | 技术选择 |
|---|---------|
| 客户端 UI | Flutter (Dart) |
| 客户端核心 | Rust（端侧推理、安全、加密、本地存储） |
| 客户端桥接 | flutter_rust_bridge (FFI) |
| 后端基础服务 | Rust |
| 后端 AI 引擎 | Python |

| 主数据库 | PostgreSQL |

| 缓存 | Dragonfly |

| 云端推理后备 | 国内大模型 API / 自建 vLLM |

## 二、客户端架构

Flutter 负责 UI，Rust 负责所有核心逻辑——AI 推理、安全、加密、本地存储。Dart 不碰 AI 逻辑。

```

│                                              │
```

## 四、数据层架构

### 4.1 数据库选型

| 数据库 | 用途 | 数据类型 |
|--------|------|---------|
| PostgreSQL | 身份、认证、治理、向量、内容、行为、心物 | 结构化数据 + pgvector + JSONB |
| SQLCipher | 端侧加密存储 | 本地敏感数据 |
| Dragonfly | 缓存 | 热数据 |
| MinIO | 对象存储 | 图片、音频、视频 |

### 4.2 数据安全

- **端侧加密**：SQLCipher 加密本地存储
- **传输加密**：TLS 1.3
- **数据脱敏**：上传前处理敏感信息
- **差分隐私**：聚合数据加入噪声

## 六、关系深度算法

### 6.1 核心原则

1. **用相对值，不用绝对值**
2. **关系深度对用户不可见**
3. **个性化基线**
4. **跨时空友好**：关系可以「暂停」而非「衰减」
5. **容量限制**：前 N% 关系获得显著曝光

### 6.2 算法实现

```python
class RelationshipDepthCalculator:
    def calculate_relative_strength(self, resonance_count, time_window, 
                                     user_a_baseline, user_b_baseline):
        """计算相对共鸣强度"""
        expected_a = user_a_baseline.avg_resonance_rate * time_window.days
        expected_b = user_b_baseline.avg_resonance_rate * time_window.days
        expected = expected_a * expected_b
        return resonance_count / expected
    
    def calculate_with_pause(self, current_depth, silence_duration, baseline):
        """关系暂停机制"""
        if silence_duration > baseline.avg_interaction_interval * 3:
            return current_depth  # 暂停，不衰减
        return current_depth
```

### 6.3 性能优化

```
事件记录（实时，轻量）→ O(1)
    ↓
关系计算（批量，异步）→ 每天凌晨
    ↓
用户查询（缓存，即时）→ O(1)
```

## 八、五层安全体系

### 8.1 安全原则

1. **端侧优先**：敏感计算在设备上完成
2. **数据最小化**：只收集必要的数据
3. **用户控制**：用户可以查看和关闭任何感知
4. **透明度**：系统行为可审计
5. **架构安全**：通过架构设计保证物理隔离，不依赖平台善意

### 8.2 五层安全体系

**第一层：硬件身份验证**
- 设备级硬件 ID 绑定，一人一号
- 硬件 ID 经不可逆变换后作为身份凭证，平台不持有原始硬件标识
- 攻击成本从「写脚本批量注册」变成「购买大量物理设备」，数量级提升

**第二层：端到端通信**
- 完全切断通过互联网端口访问中心服务器的可能
- 服务器与用户设备仅通过端到端加密通道通信
- 外部爬虫、API 滥用、中间人攻击在架构上不可行

**第三层：盲服务器处理**
- 服务器只能看到加密后的数据，无法看到用户原始表达
- 群体共识的聚合在加密状态下完成
- 即使服务器被入侵、被扣押、被内部人员接触，泄露的只有不可读的密文

**第四层：完全匿名**
- 用户之间完全匿名，无用户名、无头像、无可关联身份的任何信息
- 表达内容不可追溯到任何个体

**第五层：群体行为监测（防水军）**
- 在无搜索、无话题、无趋势的环境下，真实群体表达应呈现渐进、平稳、无突变的分布
- 突变型动向（突然高峰、突然集中）几乎必然是外部操纵，自动过滤不展示
- 真实世界事件引发的自然突变与人为操纵的区分：真实事件影响多维度表达，操纵通常集中在单一维度

### 8.3 五层防御的协同效应

```
硬件身份 → 一人一号，攻击成本极高
    ↓
端到端通信 → 外部完全隔绝
    ↓
盲服务器 → 内部也无法窥探
    ↓
完全匿名 → 用户之间无从知晓彼此
    ↓
群体监测 → 漏网操纵行为被统计过滤
    ↓
没有任何单一实体能同时看到「谁说了什么」
```
